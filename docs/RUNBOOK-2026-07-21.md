### 中线追赶的当前规则

`src/game-room.mjs` 是唯一可信的物理来源。每个 40 Hz tick：

```text
baseSpeed       = nextCameraSpeed
gap             = (cameraX + cameraAdvance + 320) - (playerX + baseSpeed × dt)
ratio           = min(1, gap / 320) × 0.75
percentMove     = baseSpeed × ratio × dt
legacyFloorMove = gap × 0.2 × dt
recoveryMove    = max(percentMove, legacyFloorMove)  # gap > 4 px
playerVx        = baseSpeed + recoveryMove / dt
```

含义：角色与镜头同速；只要角色在 320 px 屏幕中线左侧，额外速度按当前镜头速度和落后比例持续存在；越接近中线，额外速度越平滑地减小。`gap <= 4 px` 时直接对齐中线。`blockedX` 时不得穿越方块，但 `playerVx` 仍保留为上面的目标速度。

例如现场诊断曾出现“距中线 156 px、镜头约 474”：额外速度应约为 `474 × 0.75 × 156 / 320 ≈ 173`，角色目标速度约为 647，而不是 509 或 474。

## 3. 前端诊断如何解读

比赛画面中红色虚线是**固定 Canvas 屏幕坐标 320 px**，不是从玩家位置计算的伪中线。移动的是赛道和共享镜头，红线本身不应横向漂移。移动端 CSS 缩放后，它会在物理像素中看起来不在截图正中，但仍是内部 640 px stage 的 x=320。

左上诊断格式：

```text
距中线 <N>px · 角色 <vx> · 镜头 <cameraSpeed> · 追赶/受阻
```

- `追赶` 且 `N > 4`：应看到 `角色 > 镜头`。
- `受阻`：本帧前方侧墙 sweep 命中。角色不能穿块，但其 `角色` 数值仍应大于镜头（若 `N > 4`）；清开方块后应立即把位置追回来。
- 若 `N > 4`、显示 `追赶`、且角色仍等于镜头，这是新回归；优先保存截图、房间码、tick、四项诊断数值与是否刚翻转。

## 4. 网络与呈现分层

不要把三层数据混在一起：

1. **服务器物理**：`GameRoom` 计算 `x/y/vx/vy/cameraX/cameraSpeed`，决定碰撞、淘汰、分数和排名。
2. **传输**：`realtime-server.mjs` 把比赛状态压成紧凑包。比赛包会携带坐标、速度、重力与 `blockedX` 标志；不可丢弃的大厅/开始/结算消息走可靠直发路径。
3. **浏览器呈现**：`public/player-presentation.js` 与 `public/camera.js` 只在包与包之间外推，收到权威快照后平滑校正。它们不得修改胜负、碰撞或服务器坐标。

如果看见“画面抖动但服务器诊断稳定”，先检查第 3 层；如果 `state` 中 `vx` 自己就不对，先检查第 1 层；如果到包延迟/跳 tick 较大，检查第 2 层、反向代理、同机 CPU 和公网链路。

## 5. 已知边界与下一步

1. **侧墙/复杂翻转路径仍需 SWF 实机逐帧验证。** 当前是稳定的联网 sweep，不等同于 Flixel `colHullX/Y` 的全部规则。不要为了消除一个疑似空气墙而删除全部侧墙碰撞；先保存复现赛段和输入序列，再写测试。
2. **marathon 是在线产品扩展，不是原版多人六局赛制。** 原版证据和应有赛制见 `SWF_FORENSICS_ADDENDUM.md`。
3. **浏览器诊断是临时开发工具。** 追赶与碰撞稳定后，可通过环境开关隐藏红线和文字；在完全确认前不要删除，它是判断“物理慢”与“镜头假推进”的最直接证据。
4. **生产日志只用于排障。** `journalctl -u gswitch-online` 可查看服务日志；不要把原始生产日志、用户昵称、IP 或凭据提交到 Git。

## 6. 接手步骤

```powershell
git pull --ff-only
npm test
npm run test:coverage
npm start
```

先用两个浏览器窗口建立新房间，再验证：红线固定、未受阻时落后者的 `角色 > 镜头`、翻转后无穿墙、被淘汰后进入观战、结果页按服务器名次和分数显示。任何物理改动都遵循：写会失败的测试 → 提交 `test:` 检查点 → 最小修复 → 全量测试/覆盖率 → 提交 `fix:` 检查点。
