# 《反重力双英》SWF 取证续报：多人运行时、触发器与碰撞内核

> 范围：本续报只陈述 `assets/reverse/4399-44052-32.swf`（SHA-256 `D82DB5C…D645F`）中可由 AS3 导出物复核的行为。它补充 [SWF_DEEP_UNPACK_REPORT.md](./SWF_DEEP_UNPACK_REPORT.md)，不把当前联网版的长赛道、WebSocket 或服务器判定误写成原版功能。

## 1. 结论先行：原版多人是“分局生存赛”，不是一张连续终点赛道

这项结论来自 `PlayState.as` 的完整创建、结算和重开路径：

```text
创建 PlayState
  → round 加 1，循环为 1…6
  → 载入 MP02 / MP03 / MP04
  → 四名角色同时变身出场
  → 共用相机自动前进；掉队或离开上下边界即淘汰
  → 剩一人时开始 1.7 s 结算倒计时
  → 结算面板显示名次与累计胜场
  → 玩家按“下一局”后重新创建 PlayState，进入下一 round
```

`round` 与关卡是一一对应的六局循环：

| round | 原始 plist | 变身结束时相机初速度 |
|---:|---|---:|
| 1、4 | `mp02plist` | 160.3775 px/s |
| 2、5 | `mp03plist` | 153.9624 px/s |
| 3、6 | `mp04plist` | 153.9624 px/s |

证据在 `PlayState.as:914–923`、`1080–1093`、`1990–1999` 和 `Player.as:382–401`。`onNext()` 只把 `reload` 置真；随后 `PlayState.update()` 直接 `FlxG.state = new PlayState()`，并非把下一张图无缝拼到当前图右侧（`PlayState.as:3208–3227`）。

因此，当前在线版的 `MP02 → MP03 → MP04 → MP02 → MP03 → MP04` 连续 marathon 是为了联机长赛道体验做的**产品适配**，不是 1:1 原版赛制。若目标是严格复刻，应改为“每局存活排名 + 结果页 + 下一局确认”，而不是用 `finishX` 决定胜负。

## 2. 多人横向运动：相机才是唯一节拍器

### 2.1 起跑与变身门

原作创建 P1–P4 的固定初始位置、初速度和重力如下；P2/P4 紧接着调用一次 `SwitchGravity()`：

| 玩家 | 逻辑坐标 `(x,y)` | 初始水平速度 | 初始重力 |
|---|---:|---:|---:|
| P1 | `(316, 103)` | 211.6983 | 向下 |
| P2 | `(316, 325)` | 211.6983 | 向上 |
| P3 | `(316, 240)` | 211.6983 | 向下 |
| P4 | `(316, 188)` | 211.6983 | 向上 |

但构造函数把 `velocity.x` 置为 0，并以 `morph`（22 帧、20 fps）开始；所以可见赛跑不会在画面出现时立即横移。每个角色的 `morph` 到达帧 44 后才进入 `run`，并启动相机速度与加速度。名义变身时长约为 1.10 s（22 / 20）；实际显示受 Flash 动画回调所在帧影响。

来源：`PlayState.as:925–960`、`Player.as:75–105`、`Player.as:382–401`。

### 2.2 每帧强制同步，而不是四条独立加速曲线

`FlxState` 更新完所有对象后，`PlayState.update()` 对每一名多人玩家执行：

```actionscript
player.velocity.x = _camera.velocity.x;
player.acceleration.x = _camera.acceleration.x;
if (player.x >= _camera.x) player.x = _camera.x;
else player.x += (_camera.x - player.x) / 5 * FlxG.elapsed;
```

这段在 `PlayState.as:2914–2928`。含义是：

- P1 离开 `morph` 后，所有存活玩家使用同一个相机横向速度和加速度；
- 任何人都不能超过相机逻辑 X；落后者只得到一个随距离衰减的追赶修正；
- 因而“你踩在对方头上但两人仍以同样横向速度前进”是原版的预期表现，并非玩家互撞传递水平动量；
- `Player.update()` 中把 `acceleration.x` 写为 `6.9440061776` 的语句，会在随后这段多人同步中被相机的 `7.740191` 覆盖。前者不应被当成多人实际速度曲线。

相机本身使用 `7.740191 px/s²`，并由 Flixel 的半步积分更新：

```text
v' = clamp(v + a × dt)
x' = x + (v + v') / 2 × dt
```

这里 `dt = 1/40 s`。这来自 `FlxObject.update()` 的两次半增量和 `FlxU.computeVelocity()`（`FlxObject.as:185–224`、`FlxU.as:139–171`）。联网版若要追逐原版，需要让相机用此积分，且在相同仿真步结束后覆盖玩家横向速度；不能把四个 `speedX` 当成独立、长期积累的状态。

### 2.3 掉队与出界的原始淘汰判定

多人 `Player.update()` 直接调用 `kill()` 的条件为：

```text
x < camera.x − 350  OR  y > 500  OR  y < −60
```

来源：`Player.as:274–289`。最后一人尚存时，`PlayState` 设 `timeToReload = 1.7`；倒计时期间相机停止加速并按初始当刻速度的 `0.8 × dt` 减速，之后出现结果面板（`PlayState.as:2383–2570`、`2914–2949`）。这也说明原作的胜利条件是**存活排名**而不是触碰地图末端。

## 3. 输入、翻转与动画是一个明确的状态机

### 3.1 输入规则

P1 接受鼠标（仅舞台 `y > 59`）、`X` 或 `Space`；P2/P3/P4 依次接受 `M/P/Q`。输入只在 `x < 122400` 时有效。若角色贴在表面（`velocity.y == 0`）立即翻转；空中输入则覆盖 `_clickDelay = 0.35`，期间一旦落地立即翻转并额外播放落地声。

这不是队列：空中连续按键只保留最后一次 0.35 秒窗口，过期后归零。来源：`Player.as:137–190`。

### 3.2 翻转不清零竖直动量

`SwitchGravity()` 仅做以下实体变化：

1. `acceleration.y *= -1`；
2. 反转 `facing` 和目标角度 0/180；
3. 多人碰撞盒 `offset.y` 在向下时为 19、向上时为 9；
4. 播放 `switch` 和音效/烟尘。

它没有把 `velocity.y` 归零，也没有把角色移动到另一侧。角色因此先保持原动量，随后被新重力拉向另一面；`switch` 在帧 74 强制转为 `fall`，而若中途已着陆则在 60 以后帧直接吸附到 `landing`。来源：`Player.as:323–378`、`404–443`。

### 3.3 可直接实现的动画规则

| 规则 | 精确值 |
|---|---|
| `morph` | 帧 23–44，20 fps；帧 44 后 `run` |
| `run` / `runfast` | 帧率 `vx × 0.088`；阈值 `vx >= 513.208` |
| `fall` | 多人离地先延迟 4 次更新，再显示；`switch` 不会被立即覆盖 |
| `push` / `slide` | 帧率 `vx × 0.03`；在地面侧撞为 `push`，空中为 `slide` |
| `landing` | 固定 67 fps；帧 86 后回 `run` |

来源：`Player.as:96–105`、`191–272`、`404–443`，以及 `FlxSprite.changeAnimFrameRate()`（`FlxSprite.as:647–659`）。

## 4. 玩家互撞：十轮、先判纵向、再判横向

这是本次最重要的深入结论之一。原作并不使用刚体质量、承重方或“反重力把对方顶开”的单向规则，而是每个更新周期对同一玩家组连续跑 **10 次** `FlxU.overlap(_players, _players, collideMovingPlayers)`。

`collideMovingPlayers()` 的可执行规格（`PlayState.as:3797–3887`）：

1. 忽略同一对象；把双方 Y 回退一帧的 `velocity.y × elapsed`；
2. 若回退后的 `abs(y1-y2) >= height-8`（多人高度为 48，阈值为 40），按纵向接触处理；否则走横向分离；
3. 纵向接触的穿透量为 `48 - abs(y1-y2)`；
4. 若二者竖直速度反向，按各自本帧位移绝对值比例平分穿透；若同向，则把后者直接贴在前者高度之外；
5. 只要任一人的 `vy == 0`，或二者 `vy` 异号，就根据原运动方向更新 `facing`，并把**双方** `vy = 0`；
6. 若判为横向接触，左边者被放到右边者 `x - width × 1.1`（37 px 宽时约 40.7 px 间隔）；
7. 这一回调总共重复 10 次，才结束该 tick。

所以同列、重力相反的两人会互相消除竖直速度，保持同速横移；其中一方再次翻转后，会先继承它已有的 `vy`，再由下一轮互撞重新决定是否互相抵消。这与“下方玩家承重”或“反重力玩家必然顶走上方玩家”都不同。严格复刻应按上述 10 轮顺序实现，不能只做一次 AABB 推开。

## 5. 世界方块碰撞：Flixel 不是简单的轴向 sweep

`FlxU.collide(_blocks, _objects)` 先在 Quadtree 做 **X 轴**解算，再做 **Y 轴**解算（`FlxU.as:214–229`）。它以对象上一帧位移扩展 `colHullX/Y`；在每个轴上：

- 两物体该轴位移相同则不解算；
- 使用方向碰撞开关（`collideLeft/Right/Top/Bottom`）选择可碰的一侧；
- 校正量为 0 或超过任一可动对象该 hull 尺寸的 80% 时，放弃本次校正；
- 固定方块与玩家碰撞时，玩家吃完整修正；两个可动物体碰撞时，各退一半且传给 `hit*` 的对方速度也折半；
- Y 轴落在会移动的固定物体上时，额外带入其 `colVector.x`，形成平台横向携带。

80% 保护阈值特别关键：它解释了高帧延迟、翻转后偏移变化或深度相交时，原作有时会选择“本帧不硬拉回”而不是制造巨大回弹。当前在线版的脚底检测和侧墙 sweep 是稳定性适配，并不等价于这套 `colHull` 解算。完整源码依据：`FlxObject.as:185–224`、`FlxU.as:232–434`、`437–655`。

## 6. 关卡记录与触发器：多人实际只消费实体方块

每个 plist 第一数组的记录按 `integer` 类型转为如下对象（`PlayState.as:1147–1183`）：

| 记录类型 | 原始构造 | 作用域 |
|---:|---|---|
| 1 | 固定 `FlxObject(x*34, 425-y*34, 34, 34)` | 实体方块 |
| 2 | 同上，外加方块上/下各一个 `34×1` 的 type-2 overlap | 单人加速触发 |
| 3 | 固定方块，`real[2]×0.64151` 作为速度/检查点关联值 | 单人检查点网络的一部分 |
| 4 | 同坐标各建一个 type-4（玩家）与 type-5（敌人）overlap | 单人检查点/追兵速度改变 |

地图物理单位是 **34 px**，坐标变换是 `worldX = int(real[0])*34`、`worldY = 425-int(real[1])*34`；不是 48 px。图层美术对象则采用 `0.641509` 的另一套坐标换算，并且可能再按中心锚点施加 `scaleX/scaleY`，故视觉与碰撞不可混用。

更重要的是，`_objects` 在多人时只包含 `_players` 和 `_enemies`，但触发检查仍写为：

```actionscript
FlxU.overlap(_player, _overlapAreas, OverlapTile);
FlxU.overlap(_enemy, _overlapAreas, OverlapTile);
```

其中 `_player` 是多人模式额外创建、却未加入 `_objects` 的占位 Player（`PlayState.as:960`、`1622–1624`、`3145–3147`）。据此，type 2/4/5 的加速和检查点逻辑是单人流程，**本地多人真正消费的是固定 34 px 方块、共用相机和玩家互撞**。把 `finishX`、加速板或单人检查点直接加入多人判胜，是联网版的设计选择，不是原版多人行为。

单人 type-2 的加速量也已解出，触发一次为：

```text
v ← v + (-0.05 × min(v, 769.812) + 51.3208)
```

并播放加速 loop；另一个 `onSpeed()` 入口使用 `-0.0320755 × v + 51.3208`。两者不要混为同一个速度公式（`PlayState.as:3428–3450`、`3462–3541`）。

## 7. 可复核的赛道规模（仅实体方块）

当前从嵌入数据解析出的原始碰撞列表可复算为：

| 图 | 固定方块数 | 最大 X 单元 | 34 px 几何宽度 |
|---|---:|---:|---:|
| MP02 | 1,017 | 470 | 16,048 px |
| MP03 | 1,062 | 496 | 16,932 px |
| MP04 | 1,150 | 497 | 16,966 px |

这些数用于审计原始数据规模，不能据此推断原版多人存在“到达最右端就完成”的判定；其结算仍由第 1 节的淘汰逻辑驱动。

## 8. 对联网复刻的更新后规格

若追求“原版规则优先”，下一轮实现应按以下顺序：

1. 把在线房间改为一局一个 MP 关卡；服务器保留 6 局循环和累计胜场，而不是跨图拼接与 `finishX`。
2. 用相机作为唯一横向速度源：morph 后启动对应关卡相机初速度；每个 40 Hz tick 用半步积分，并在玩家更新后覆写其横向速度。
3. 实现 0.35 s 翻转缓冲、9/19 px 碰撞偏移、`switch → fall/landing` 回调和速度驱动动画。
4. 将玩家互撞替换为“回退判别 + 10 轮 callback”的同序算法；先用两窗口录制对比所有同重力/反重力组合。
5. 最后再决定是否保留 marathon、终点、服务器排名等在线扩展；它们应在 README 中明确标为“联机扩展规则”。

尚未由源码单独证明的部分是 Flash 运行时在极低帧率、浏览器渲染丢帧或网络延迟下的表现；这些必须通过原 SWF 实机帧录制验证，不能用本报告替代。
